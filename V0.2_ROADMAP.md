# v0.2 生产就绪路线图

## 目标

将 connect-ip-tunnel 升级为可用于生产环境的基础设施组件，并为开源版/企业版分层奠定基础。

## 核心改进清单

### A. 可观测性（Observability）

#### A1. 结构化日志
**目标**：从 `log.Printf` 升级到结构化日志，支持远程日志收集

**实现方案**：
```
log 包选择：github.com/charmbracelet/log 或 slog（Go 1.21+）
日志级别：DEBUG, INFO, WARN, ERROR
结构化字段：
  - request_id：追踪单个请求
  - session_id：追踪会话生命周期
  - client_cert_cn：记录客户端身份
  - duration_ms：操作耗时
  - error：错误堆栈（仅 ERROR 级别）
```

**关键日志点**：
- 连接阶段：[INFO] session_established session_id=xxx client=CN=xxx
- 异常阶段：[WARN] packet_drop session_id=xxx reason=channel_full
- 错误阶段：[ERROR] allocate_ip_failed error=pool_exhausted

**开源版/企业版区别**：
- 开源版：日志输出到 stdout/file
- 企业版：支持 syslog / CloudWatch / DataDog / ELK 集成（插件化）

### B. 多 Session 并行（方向4）

#### B1. 客户端多 Session 架构

**当前状态**：
```
Engine
  └─ connectAndRun()
      └─ 单个 CONNECT-IP session
          └─ PacketPump (TUN → session → 服务端)
```

**目标状态**：
```
Engine
  └─ runLoop()
      └─ MultiSessionPool (维护 N 个 session)
          ├─ session[0]
          ├─ session[1]
          └─ session[N]
      └─ PacketDistributor (按 flow hash 分发)
          └─ 向各 session 分发上行包
      └─ PacketAggregator
          └─ 汇聚各 session 下行包 → TUN
```

**设计细节**：

```go
// engine/multi_session.go

    sessions       []*connectip.Session
    n              int                    // session 数量
    distributor    *FlowDistributor       // 上行包分发
    aggregator     *PacketAggregator      // 下行包汇聚
    sessionMu      sync.RWMutex
    reconnectPools []map[string]time.Time // 各 session 的重连状态
}

type FlowDistributor struct {
    // 按五元组哈希分发
    // src IP + dst IP + protocol → session index
}

// 按目标 IP/协议计算哈希，避免乱序
func (f *FlowDistributor) SelectSession(pkt []byte) int {
    hash := computeFlowHash(pkt)
    return hash % len(f.sessions)
}

type PacketAggregator struct {
    // 从 N 个 session 的 inbound channel 汇聚
    // 合并到一个 TUN 写 channel
}
```

**连接管理**：
```
建立时：
  1. 计算需要的 session 数 N = min(cfg.NumSessions, num_cpus)
  2. 并发建立 N 个 CONNECT-IP session
  3. 注册到 MultiSessionPool

运行时：
  4. PacketDistributor 按 flow hash 选 session
  5. PacketAggregator 汇聚多个 inbound channel

断开重连时：
  6. 失败的 session 独立重连（不影响其他 session）
  7. 若所有 session 都断开，���个 runLoop 失败
```

**优点**：
- CPU 多核利用率提升 N 倍
- 单 session 连接数限制绕过（单 QUIC 连接 stream 数有上限）
- 相同 flow 的包同一 session，避免乱序

**实现复杂度**：中（关键是 flow hash 和 aggregator 的设计）

**开源版/企业版区别**：
- 开源版：支持 N=1（单 session 模式）
- 企业版：支持 N=1~64（多 session 并行，可按 CPU 自动调整）

#### B2. 性能对标

**Benchmark 目标**：

```
对比对象：WireGuard (Linux kernel)

场景1：单流（单 TCP 连接）
  WireGuard：~900 Mbps (kernel bypass)
  connect-ip-tunnel v0.1：~600 Mbps (user-space HTTP/3)
  connect-ip-tunnel v0.2 (8 sessions)：~950 Mbps+ (接近线速)

场景2：多流并行（16 并发）
  WireGuard：~900 Mbps (单核饱和)
  connect-ip-tunnel v0.1：~300 Mbps (单核 QUIC 处理)
  connect-ip-tunnel v0.2 (8 sessions)：~7.2 Gbps (8 核 × 900 Mbps)

场景3：延迟（ping over tunnel）
  WireGuard：~0.2ms (kernel)
  connect-ip-tunnel v0.2：~2-3ms (user-space + HTTP/3 serialization)
```

**测试代码位置**：`engine/multi_session_benchmark_test.go`

#### C2. 配置热重载

**目标**：更新 JSON 配置文件后，不需要重启服务

**实现方案**：

```go
// option/watcher.go
type ConfigWatcher struct {
    filePath string
    interval time.Duration
    callback func(*Config) error
}

func (w *ConfigWatcher) Watch(ctx context.Context) {
    ticker := time.NewTicker(w.interval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            newCfg, err := LoadConfig(w.filePath)
            if err != nil {
                log.Warn("failed to reload config", "error", err)
                continue
            }
            if newCfg != w.lastCfg {
                if err := w.callback(newCfg); err != nil {
                    log.Error("config update failed", "error", err)
                } else {
                    log.Info("config reloaded successfully")
                }
            }
        }
    }
}

// server/server.go
func (s *Server) setupConfigWatch(ctx context.Context) {
    watcher := &option.ConfigWatcher{
        filePath: s.cfg.ConfigPath,
        interval: 10 * time.Second,
        callback: func(newCfg *option.ServerConfig) error {
            // 更新关键参数（IP 池、URI template 等）
            return s.applyConfigUpdate(newCfg)
        },
    }
    go watcher.Watch(ctx)
}
```

**可热重载的参数**：
- IP 池范围（IPv4/IPv6）
- URI template
- TLS 证书路径（提前加载新证书，选择合适时机切换）
- 日志级别
- Metrics 采样率

**不可热重载的参数**（需要重启）：
- 监听端口
- QUIC 参数（buffer size 等）

### D. 证书热重载

#### D1. TLS 证书热重载

**当前问题**：证书过期或更新需要重启服务

**解决方案**：

```go
// security/tls/cert_manager.go
type CertManager struct {
    certPath   string
    keyPath    string
    reloadCh   chan struct{}
    currentCrt *tls.Certificate
    mu         sync.RWMutex
}

// 后台定期检查证书有效期
func (m *CertManager) WatchCertValidity(ctx context.Context) {
    ticker := time.NewTicker(1 * time.Hour)
    defer ticker.Stop()
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            m.checkAndReload()
        }
    }
}

// 主动触发重载（运维发现证书更新后调用）
func (m *CertManager) Reload() error {
    crt, err := tls.LoadX509KeyPair(m.certPath, m.keyPath)
    if err != nil {
        return err
    }
    m.mu.Lock()
    m.currentCrt = &crt
    m.mu.Unlock()
    log.Info("certificate reloaded")
    return nil
}

// TLS 握手时读取最新证书
func (m *CertManager) GetCert() *tls.Certificate {
    m.mu.RLock()
    defer m.mu.RUnlock()
    return m.currentCrt
}
```

**集成到 TLS 配置**：
```go
tlsCfg := &tls.Config{
    GetCertificate: func(hello *tls.ClientHelloInfo) (*tls.Certificate, error) {
        return certMgr.GetCert(), nil
    },
}
```

**开源版/企业版区别**：
- 开源版：手动触发 HTTP API 重载
- 企业版：自动定期检查 + 证书即将过期告警 + ACME 自动续期支持
